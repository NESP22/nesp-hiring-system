# NESP Hiring System — Render Deployment Guide

## Status

This guide prepares a Render Blueprint deployment. It does not authorize deployment, billing, public job posting, email sending, or real applicant uploads.

The Blueprint creates two new resources only:

1. `nesp-hiring-web` — OpenCATS web service
2. `nesp-hiring-db` — private MariaDB service

Existing Render services such as the NESP email assistant, phone-ticket reconciliation, and monitor database must remain untouched.

## Safety design

- MariaDB is a private Render service, not a public web service.
- Database passwords are randomly generated by Render.
- The database has a persistent disk at `/var/lib/mysql`.
- OpenCATS has a separate persistent disk at `/var/data` for configuration, installer lock, resumes, uploads, attachments, and related state.
- The complete OpenCATS site is protected by temporary Basic Authentication during installation and testing.
- The health-check file remains public so Render can monitor the service.
- OpenCATS outbound email is disabled during initial testing.
- Auto-deploy occurs only after GitHub checks pass.

## Before approving the Blueprint

Confirm all of the following:

- Repository: `NESP22/nesp-hiring-system`
- Blueprint file: `render.yaml`
- Branch: `master`
- Region: Virginia for both new services
- No existing service is being modified or deleted
- The pricing screen shows only the two new services and their two disks
- The installer password is entered privately and is not reused anywhere else

Do not use a password from email, banking, hosting, Cloudflare, GitHub, or another NESP system.

## Phone deployment steps

After the Render deployment PR is reviewed, tested, and merged:

1. Open the Render dashboard.
2. Open the main menu.
3. Choose **New**.
4. Choose **Blueprint**.
5. Connect or select GitHub repository `NESP22/nesp-hiring-system`.
6. Confirm Render detects the root `render.yaml` file.
7. Review the two proposed services carefully.
8. For `INSTALL_ACCESS_PASSWORD`, enter a new random password of at least 20 characters.
9. Save that password in the phone password manager. Do not send it in chat or screenshots.
10. Review the exact monthly price shown by Render.
11. Do not approve the deployment if Render shows any unexpected deletion, rename, or change to existing services.
12. Approve only when the resource list and cost are understood.

## Initial private test

After both services report healthy:

1. Open the `nesp-hiring-web` Render URL.
2. Sign in to the temporary setup gate with:
   - Username: `nesp-admin`
   - Password: the private value entered during Blueprint creation
3. Confirm that the OpenCATS installation screen appears.
4. Do not upload real resumes.
5. Use fake applicant names and fake documents during testing.
6. Keep OpenCATS email disabled.

## OpenCATS installation

The web container automatically receives the database hostname, database name, database user, and generated password through Render's private references. The persistent `config.php` is updated from those environment values whenever the service starts.

The installation wizard may still display database fields. Use the values stored on the `nesp-hiring-web` Environment page:

- Database name: `cats`
- Database user: `opencats`
- Database host: value of `DB_HOST`
- Database password: value of `DB_PASSWORD`

Do not show these values in screenshots.

During the installer:

- Resume text tools should resolve to the Linux paths available in the container.
- Keep application email disabled or select the no-mail option.
- Create a unique OpenCATS administrator username and password.
- Confirm installation completes and creates the persistent installer lock.

## Acceptance test before real applicants

The system is not production-ready until all tests pass:

- Administrator can sign in after a restart or redeploy.
- A fake job can be created.
- A fake applicant can submit a fake resume.
- The fake resume remains available after a restart.
- Candidate search works.
- Database data remains after a restart.
- An unauthenticated visitor cannot bypass the temporary password gate.
- OpenCATS sends no email.
- Existing Render services remain healthy and unchanged.
- A database dump is created and copied to an encrypted off-platform location.

## Backup gate

Render creates daily disk snapshots, but Render specifically recommends logical dumps such as `mysqldump` for a custom MySQL/MariaDB database instead of restoring its raw disk snapshot.

Before real applicant data is accepted, configure and test:

- Daily MariaDB logical dump
- Encrypted off-platform storage
- Applicant-file backup from `/var/data`
- Retention policy
- Restore test using an isolated service

## Going public later

Do not remove the temporary password gate until:

- The public applicant portal path is identified and tested.
- The administrative area has a separate protection plan.
- Cloudflare domain and access rules are reviewed.
- Privacy notice and applicant consent language are approved.
- Job postings and compensation are approved.
- AI screening remains advisory and human-reviewed.
- Email templates and sending permissions are approved.

## Rollback

If the new services fail before real applicant data exists:

1. Do not change any existing Render service.
2. Suspend or delete only `nesp-hiring-web` and `nesp-hiring-db`.
3. Confirm both new disks are no longer generating charges if the project is abandoned.
4. Keep the GitHub repository and test records for review.
